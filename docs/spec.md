# Typing Level Zero 仕様書

## ステータス

実装レビュー前のドラフト。

## 背景

Typing Level Zeroは、GitHub Pagesで動作する静的なタッチタイピング練習アプリである。アカウント、サーバー、外部APIを必要とせず、集中して練習できる入力画面と、ローカルデータを使った分析画面を提供する。

## 目的

10文字のランダムな文字列を1 playとして入力する練習体験を提供する。入力速度、正確性、リセット、キーごとの苦手傾向を分析できるだけの詳細なローカルデータを保存する。

## 対象範囲

初期リリースには以下を含める。

- ViteとReactによるWebアプリ
- 常に表示される10文字のランダムな小文字アルファベット
- 大文字・小文字を区別しない入力判定
- タイプミス時に文字列の先頭へ戻る挙動
- キー単位・入力単位の時間計測
- 完了したplayをすべて保存するIndexedDB永続化
- `/analysis` に配置する全体分析ページ
- `/keys` に配置するキー別分析ページ
- `/history` に配置するplay履歴ページ
- GitHub ActionsによるGitHub Pagesへの静的デプロイ

## 対象外

初期リリースでは以下を扱わない。

- ユーザーアカウントやサーバー側ストレージ
- マルチプレイや共有ランキング
- 制限時間や強制タイムアウト
- 単語辞書や言語別のタイピングカリキュラム
- キーボード入力の代替となるモバイルタッチ操作

## 用語

- **play**: 1つの10文字プロンプトを完了するまでの1回の練習単位
- **prompt**: 生成されて表示される10文字の小文字アルファベット列
- **cursor**: 次に入力する文字のインデックス
- **reset**: タイプミス後にcursorを0へ戻すこと
- **完了したplay**: プロンプトの10文字すべてを正しく入力したplay

## 機能要件

### 練習フロー

1. アプリは常に10文字のpromptを表示する。
2. promptは`a`から`z`までの小文字ASCIIアルファベットで構成する。
3. 隣り合う文字は同じにしない。隣り合っていない位置では同じ文字を許可する。
4. 開始ボタンやSpaceキーを必要としない。
5. 最初のアルファベットキー入力でplayを開始する。その入力はゲーム上のcursor更新には使うが、分析上は完全に無視する。
6. アルファベットの入力は大文字・小文字を区別しない。たとえば、期待値が`a`のとき`A`も正解とする。
7. 正しい入力ではcursorを1つ進める。
8. 2回目以降の間違った入力ではミスを記録し、reset回数を増やし、同じpromptを維持したままcursorを0へ戻す。playの最初の入力が間違っていた場合は、ミスやresetとして扱わず、cursorを変更しない。
9. resetではplayを終了しない。promptを完了するか、playをキャンセルするまで継続できる。
10. 10文字すべての入力が完了したら現在のplayを終了し、結果を保存して、即座に新しい10文字のpromptへ差し替える。
11. 次のplayはpromptを表示した待機状態になり、次のアルファベットキー入力で開始する。
12. 制限時間は設けない。ただし分析用に経過時間を計測する。

### 視覚的フィードバック

入力状態は、別の説明を読まなくても分かるようにprompt自体で表現する。

- 入力済みの文字は、薄いグレーと少し透明なスタイルにする。
- 現在入力する文字、未入力の文字、入力済みの文字を明確に区別する。
- reset後も現在位置をすぐに識別できるようにする。
- ミス時にエラーが分かるフィードバックを表示する。
- 位置は色だけに依存せず、別の視覚的な手掛かりでも区別できるようにする。

### 永続化

1. 完了したplayをIndexedDBに保存する。
2. ユーザーが明示的に履歴を削除しない限り、完了したplayをすべて保持する。
3. 未完了またはキャンセルされたplayは、完了結果として保存しない。
4. 保存データはブラウザと端末内に限定する。初期リリースではサーバーへ送信しない。
5. 分析ページは、ハードコードまたはメモリ上だけのデータではなくIndexedDBからplayを読み込む。

### 保存するplayデータ

完了したplayごとに、集計データと入力イベントの生データを保存する。論理的な構造は以下とする。

```ts
interface PlayRecord {
  id: string;
  prompt: string;
  startedAt: number;
  completedAt: number;
  durationMs: number;
  summary: {
    totalInputs: number;
    correctInputs: number;
    mistakeCount: number;
    resetCount: number;
    accuracy: number;
    grossCpm: number;
    grossWpm: number;
    netCpm: number;
    netWpm: number;
  };
  events: InputEventRecord[];
}

interface InputEventRecord {
  sequence: number;
  timestampMs: number;
  intervalMs: number | null;
  positionBefore: number;
  positionAfter: number;
  expectedKey: string;
  actualKey: string;
  physicalCode: string;
  isCorrect: boolean;
  resetCount: number;
}
```

実際のTypeScript型では、より狭い型やreadonlyプロパティを使用してよい。ただし保存情報はこの構造と同等にする。

アルファベット入力の`actualKey`は分析用に小文字へ正規化する。一方、`physicalCode`はブラウザのキーボードコードを保持する。これにより、論理キーのミスと物理キーボード上の位置を区別して分析できる。

playの最初のアルファベット入力は`events`に保存しない。`summary.totalInputs`にだけ含め、正解数、ミス数、reset数、正答率、速度、latencyには含めない。最初の入力後に保存される最初のイベントの`intervalMs`は`null`とし、直前の保存済み入力が存在しないイベントとして扱う。

### 全体分析ページ

`/analysis`ページでは、練習全体の概要と推移を確認できるようにする。以下を表示する。

- 完了したplayの総数
- 平均完了時間と最速完了時間
- 平均正答率
- 平均のgross CPM/WPMとnet CPM/WPM
- 総ミス数と総reset数
- 最近の完了時間、正答率、速度の推移
- 文字列の1〜10文字目ごとの分析
- キー別分析ページ(`/keys`)への導線
- play履歴ページ(`/history`)への導線
- 完了したplayがない場合の空状態
- ローカルに保存した履歴を全削除する、確認付きの操作

分析結果は入力イベントの生データと保存済みplayから再集計する。表示値と保存データの整合性を保てるようにする。

### プレイ履歴ページ

`/history`ページでは、完了したplayを一覧で確認できるようにする。

- 完了日時の新しい順にplayを表示する。
- prompt、日時、完了時間、速度、正答率、ミス数、reset数を表示する。
- playを選択すると、そのplayの詳細データを確認できるようにする。
- 完了したplayがない場合は空状態を表示する。
- `/analysis`とメインプレイページへ戻る導線を用意する。

### キー別分析ページ

`/keys`ページでは、キーごとの入力状況をランキングと詳細情報で確認できるようにする。

#### ランキング

- **苦手キーランキング**: 正答率が低い順に表示する。
- **ミスが多いキーランキング**: ミス数が多い順に表示する。
- **入力が遅いキーランキング**: キー入力の平均または中央値のlatencyが長い順に表示する。
- **得意キーランキング**: 正答率が高く、latencyが短い順に表示する。

各ランキングには、対象キー、入力回数、正解数、ミス数、正答率、ミス率、平均latency、中央値latencyを表示する。入力回数が少ないキーは、ランキング結果だけで判断できないことが分かるようにサンプル数を併記する。

#### キーごとの詳細

キーを選択すると、そのキーを期待値として入力したときの詳細を表示する。

- 対象キーの入力回数、正解数、ミス数、正答率、ミス率
- 対象キーの平均latency、中央値latency、最速latency、最遅latency
- 対象キーを入力した位置ごとの正答率とlatency
- 対象キーで発生したミスの一覧
- 対象キーの代わりに押したキーのランキング
- 「期待キー → 実際に押したキー」の組み合わせごとの回数と割合
- 対象キーの成績推移

たとえば期待値が`a`の場合、`a`の入力成績だけでなく、`a`の代わりに`s`や`q`などを何回押したかも確認できるようにする。キーごとの詳細から、他のキーの詳細へ移動できるようにする。

キーごとの状態を一覧できるキーボードレイアウト表示も`/keys`に配置し、正答率、ミス数、入力回数、latencyなどの指標を切り替えられるようにする。`/analysis`には概要のみを表示し、キー単位の分析は`/keys`へ集約する。

## 指標の定義

- **Duration**: 分析対象として保存される最初の入力から、10文字目を正しく入力するまでの時間
- **Accuracy**: 正しいアルファベット入力数を、判定対象となったアルファベット入力総数で割った値
- **CPM**: playの時間と対象入力数から算出する1分あたりの文字数
- **WPM**: 5文字を1単語として算出する1分あたりの単語数
- **Gross speed**: ミスを含むすべての判定対象入力から算出した速度
- **Net speed**: 記録されたミス数に応じてGross speedを減算調整した速度
- **Key latency**: 現在の入力イベントと直前の保存済み入力イベントとの時間差。playの最初の入力イベントは対象外とする
- **Reset count**: cursorを0へ戻したミスの回数

指標の計算式は、練習画面の結果表示と分析ページで統一する。計測時間が0の場合は、無限大や`NaN`を表示しない。

### 最初の入力の扱い

- 各playの最初のアルファベット入力は、playを開始するための入力として扱う。
- 最初の入力がpromptの先頭文字と一致した場合はcursorを1つ進めるが、分析用データには保存しない。
- 最初の入力が間違っていた場合は、入力回数にだけ含める。ミス、reset、latency、速度には含めず、cursorも変更しない。
- 最初の入力は、正誤に関係なく`summary.totalInputs`へ1回分だけ加算する。
- 最初の入力後に行われる入力だけを分析対象として保存する。
- reset後の入力イベントは、直前の保存済み入力イベントからの時間を計測する。

## IndexedDB設計制約

- 将来のスキーマ移行に備えて、バージョン付きのIndexedDBデータベースを使用する。
- play IDをキーとする`runs`オブジェクトストアを使用する。
- 入力イベントは完了したplayと同じレコードに保存し、1回の書き込みで原子的に保存する。
- アプリケーション側で履歴件数の上限を設けない。
- ストレージが利用できない場合や書き込みに失敗した場合は、原因と対処を画面上に表示する。
- 分析計算は、保存されたレコードから決定的に再現できるようにする。

## デプロイ要件

- アプリは静的なViteバンドルとしてビルドする。
- リポジトリ名以下で配信されるGitHub Pagesのベースパスに対応する。
- 指定ブランチからビルドしてGitHub PagesへデプロイするGitHub Actionsワークフローを用意する。
- `/analysis`、`/keys`、`/history`のすべてをGitHub Pages上から開けるようにする。

## 受け入れ条件

- アプリを開くと、Spaceを押さなくても10文字のpromptが表示される。
- 最初のアルファベット入力で計測が始まり、その入力が即座に判定される。
- 正しい入力で10文字分の視覚的なcursorが進む。
- 間違った入力がエラーとして表示され、同じpromptの1文字目へ戻る。
- 10文字を完了すると、IndexedDBにplayレコードが1件だけ保存される。
- 完了直後に新しいpromptが表示される。
- 各playに、最初の入力を除いたキー、期待値、位置、physical code、正誤、時間の生データが含まれる。`summary.totalInputs`だけは最初の入力を含むため、保存された`events`の件数と一致しない場合がある。
- `/analysis`にIndexedDB由来の全体集計、位置別、推移、play別の情報が表示される。
- `/keys`に苦手キー・得意キーのランキングと、キーごとの誤入力先の情報が表示される。
- `/keys`で任意のキーを選択すると、そのキーの入力回数、正答率、latency、誤入力先、成績推移を確認できる。
- `/history`に完了したplayの履歴が新しい順で表示される。
- `/history`で任意のplayを選択すると、そのplayの詳細を確認できる。
- ページを再読み込みしても、完了済みplayの履歴が保持される。
- GitHub Pagesのリポジトリベースパス配下でアプリが動作する。

3. 保存した生データを初期リリースからJSONとしてエクスポート可能にするか。
